## 1. Conventions

| Convention | Detail |
|------------|--------|
| Naming     | `snake_case` for tables and columns. Prisma's model names are `PascalCase` and use `@map` to bind to the snake_case tables. |
| Primary keys | `id uuid` (UUIDv7 generated by `gen_random_uuid()` or app-side via `uuid` v7 npm package). UUIDv7 is monotonic, friendly to B-tree indexes. |
| Foreign keys | `<table>_id uuid` for clarity. Constraint name `<child>_<parent>_fk`. `ON DELETE` is `RESTRICT` for financial data, `CASCADE` only for child rows whose existence depends on the parent (e.g., `journal_entry_line` cascades from `journal_entry`). |
| Timestamps | `created_at`, `updated_at` (`timestamptz NOT NULL DEFAULT now()`). `deleted_at timestamptz NULL` for soft delete. |
| Money      | `numeric(18,4) NOT NULL` (never nullable; default `0`). `currency char(3) NOT NULL DEFAULT 'USD'` on monetary tables. |
| Accounting date | `recognize_date date NOT NULL` on every transactional table â€” the controlling date for period gating and reporting. |
| RLS        | Every company-scoped table has `company_id uuid NOT NULL` + an RLS policy. Every account-scoped table has `tcharts_account_id uuid NOT NULL` + an RLS policy. See ADR-0003. |
| Audit columns | `created_by uuid NULL`, `updated_by uuid NULL` on every mutable table. `null` only allowed for system-inserted rows. |

### `journal_entry_line`

| Column                | Type             | Notes                                       |
|-----------------------|------------------|---------------------------------------------|
| `id`                  | `uuid` PK        |                                             |
| `journal_entry_id`    | `uuid NOT NULL FK CASCADE` |                                   |
| `chart_of_account_id` | `uuid NOT NULL FK` |                                           |
| `debit`               | `numeric(18,4) NOT NULL DEFAULT 0` |                       |
| `credit`              | `numeric(18,4) NOT NULL DEFAULT 0` |                       |
| `description`         | `text NULL`      |                                             |
| `line_order`          | `int NOT NULL`   | Display order                               |

**DB CHECK:** `(debit = 0 AND credit > 0) OR (debit > 0 AND credit = 0)` â€” a line is either a debit or a credit, never both, never neither.

**Sum-zero invariant** enforced by a deferrable trigger that runs at transaction commit:

```sql
CREATE CONSTRAINT TRIGGER je_balanced
AFTER INSERT OR UPDATE ON journal_entry_line
DEFERRABLE INITIALLY DEFERRED
FOR EACH ROW EXECUTE FUNCTION assert_journal_entry_balanced();
```

The trigger sums debits and credits for the line's `journal_entry_id` and raises if they differ by more than `0.0000`. Deferred-to-commit so multi-line inserts can pass through atomic transactions.

**Indexes:**
- `(journal_entry_id, line_order)`.
- `(chart_of_account_id, journal_entry_id)` â€” used by reports.

## 8. Revenue / Expense / Banking â€” table shapes

Each transactional table follows the **transactional table contract**:

- `id uuid PK`, `tcharts_account_id`, `company_id`, `recognize_date date NOT NULL`, `posted_at timestamptz`.
- Line-item child table with cascade delete.
- `status` enum reflecting the transaction lifecycle.
- A pointer to the originating `journal_entry` (or `journal_entries` if multiple â€” e.g., bill + bill payment).
- Soft-deletable for unposted drafts; **never** hard-deletable for posted transactions.

Full schemas are added in the sprint that ships the module (S11 for revenue, S12 for expense, S14 for banking).

## 9. Reporting â€” `mv_account_period_balances`

A materialized view to make Balance Sheet, P&L, and Trial Balance fast.

```sql
CREATE MATERIALIZED VIEW mv_account_period_balances AS
SELECT
  je.tcharts_account_id,
  je.company_id,
  jel.chart_of_account_id AS account_id,
  date_trunc('month', je.recognize_date)::date AS period_month,
  SUM(jel.debit)  AS total_debit,
  SUM(jel.credit) AS total_credit,
  SUM(jel.debit - jel.credit) AS net_debit
FROM journal_entry je
JOIN journal_entry_line jel ON jel.journal_entry_id = je.id
WHERE je.reversed_at IS NULL
GROUP BY je.tcharts_account_id, je.company_id, jel.chart_of_account_id, period_month;

CREATE UNIQUE INDEX ON mv_account_period_balances (company_id, account_id, period_month);
CREATE INDEX ON mv_account_period_balances (tcharts_account_id, period_month);
```

Refreshed:
- Nightly by a worker job (`reports.materialized-view.refresh`).
- On-demand after `period.soft-closed` and `period.hard-closed` events.
- On-demand after the first `transaction.posted` event in any 30-second window (debounced).

**RLS:** must be enforced via security barrier (`CREATE MATERIALIZED VIEW ... WITH (security_barrier=true)` is not supported on MVs â€” we use a `SECURITY DEFINER` view wrapping the MV with the same RLS predicate).

## 11. RLS policy bootstrap

Every company-scoped table gets this policy via a SQL migration that follows the Prisma migration in the same migration directory (`migration.sql` + `rls.sql`).

```sql
-- For each company-scoped table $T:
ALTER TABLE $T ENABLE ROW LEVEL SECURITY;
ALTER TABLE $T FORCE ROW LEVEL SECURITY;

CREATE POLICY ${T}_company_isolation ON $T
  USING (company_id = current_setting('app.current_company_id', true)::uuid);

-- Note: `, true` makes current_setting return NULL if unset, which (NULL = uuid) â†’ false â†’ row hidden.
-- This is the safety we want: an unset session var means zero rows visible.
```

Account-scoped tables use:

```sql
CREATE POLICY ${T}_account_isolation ON $T
  USING (tcharts_account_id = current_setting('app.current_tcharts_account_id', true)::uuid);
```

The Prisma extension `rls-context` (in `packages/db/src/extensions/rls-context.ts`) executes `SET LOCAL app.current_company_id = '...'` at the start of every transaction the app opens, using values from the request-scoped `TenantContext`.

## 12. Indexing strategy

Every company-scoped query is fast because **every index leads with `company_id`** (or `tcharts_account_id` for account-scoped tables). Reads that aren't tenant-leading are forbidden by code review.

Partial indexes for soft-delete columns:

```sql
CREATE INDEX ON $T (company_id, status) WHERE deleted_at IS NULL;
```

## 13. Migration policy

- Prisma migrations live under `packages/db/prisma/migrations/`.
- Each migration directory contains `migration.sql` (Prisma-generated) and may contain `rls.sql` (hand-written for RLS DDL Prisma can't express).
- CI runs `prisma migrate dev` against an ephemeral Postgres in test, then runs `prisma migrate diff` to verify the migration is reversible.
- Production migrations run as a one-off ECS task in the deploy pipeline, before API tasks are updated.
- **Two-phase deploy required for destructive changes** (DROP column/table): deploy the code that no longer reads the column first, then drop the column in the next sprint.
- Long migrations (>30s estimated) require a maintenance window communication.
